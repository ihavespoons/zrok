# Orchestrator Refactor Plan

Three coupled changes aimed at lowering the LLM-tool-calling-competence floor so cheaper / smaller models can produce useful eval scores:

1. **Deterministic dispatcher** — replace the orchestrator-LLM with Go code that reads a manifest and dispatches subagents in parallel.
2. **Per-tool exemplars** — every place we tell an agent to call a `zrok` command or the Task tool, ship a complete copy-pasteable example.
3. **Validate + one-shot retry** — when a subagent run fails recoverably (schema error, malformed YAML, parser error), re-invoke once with a corrective prompt.

## Why

Eval variance across models tells the story:

| Model | F1 | Failure mode |
|---|---|---|
| DeepSeek v4 Flash | 0.06 | Tool calls flaky, subagents under-dispatched |
| Gemma 4 31B IT | 0.00 (crash) | `SchemaError` — omitted `description` from Task tool call |
| DeepSeek v4 Pro | 0.40 | Works fine |
| Sonnet 4.6 | (prod) | Reliable |

The cliff isn't *code review skill* — it's *agentic tool-call reliability*. Small models break at the orchestrator level (deciding which agents to dispatch, how to format Task tool calls, when to skip SAST triage) before they ever produce a finding. The eval, as architected, partly measures harness-navigation rather than bug-finding.

## Recommended order: #2 → #1 → #3

- **#2 first** is mechanical and helps both runners (opencode and Claude Code SKILL) immediately. Ship nothing else and weak-model F1 likely lifts.
- **#1 second** is the structural fix. Doing it after #2 means the new dispatcher invokes prompts that already have exemplars baked in — no re-touch.
- **#3 lives inside #1.** Doing #3 first forces retry logic into an opencode wrapper (a black box). After #1 it lives in our Go code where it's unit-testable.

## Change #2: Per-tool exemplars

### Inventory of touch sites

- `cmd/review.go` — `renderOpenCodeOrchestrator` (deep) and `renderOpenCodeOrchestratorFast`. Has `taskToolSchemaGuidance()` already; needs exemplars for `zrok finding list`, `zrok rule add`, `zrok exception add`.
- `internal/agent/configs/agents/*.yaml` — 13 analysis agents. Each has an identical "Filing Protocol (MANDATORY)" block with `zrok finding create --title ...` placeholders. Replace `...` with a concrete shell-pasteable example.
- `skills/code-review/SKILL.md` — Claude Code path. Has Task-tool schema; needs zrok-command exemplars in the same place.
- `cmd/finding.go` — `findingCreateCmd.Long` already has a YAML example; add a flag-mode example with all required fields.

### Design decision: centralize exemplars

Create `internal/agent/tool_examples.go` with exported functions:

```go
FindingCreateExample() string         // flag-mode, all required fields, one concrete CWE
FindingCreateYAMLExample() string
FindingListExample() string
FindingUpdateNoteExample() string
RuleAddExample() string
ExceptionAddFingerprintExample() string
ExceptionAddPathGlobExample() string
TaskToolExample() string              // both Claude-Code and opencode forms
```

Single source of truth — no drift between orchestrator prompt, SKILL.md, and 13 agent YAMLs.

### Wiring into agent YAMLs (preferred option)

Add `FindingCreateExample`, `FindingListExample`, etc. to `PromptData` in `internal/agent/prompt.go`. Replace the prose `zrok finding create --title ... --severity ...` paragraph in every analysis agent's `prompt_template` with `{{.FindingCreateExample}}`. Mechanical and grep-able.

### Anti-bitrot tests

Add `internal/agent/prompt_test.go` asserting every analysis-phase agent's rendered prompt contains `zrok finding create --title`. New analysis agents will fail this until they include the exemplar. Add a stricter test that asserts each example shell-parses correctly (dry-run flag) to catch stale-form drift.

### Runner impact

- **opencode:** subagent files in `.opencode/agents/*.md` are regenerated by `zrok review pr setup`, so the change propagates automatically once `Generate()` learns the new template fields.
- **Claude Code SKILL:** manual edit to embed the same exemplars (use the same `tool_examples.go` strings literally in markdown; drift is acceptable because SKILL.md is reviewed by hand at update time).
- **Dogfood action `diffsec/zrok-review-action`:** no change for #2.

## Change #1: Deterministic dispatcher

### Design: new `zrok review pr run` subcommand

Lives in `cmd/review.go` as a sibling to `reviewPrSetupCmd` and `reviewPrReportCmd`. Reuses existing infra (project loading, store). Exposed via the same `zrok review pr` group.

Go subcommand over bash because: matches the constraint preference; goroutines + `os/exec.Command` covers parallelism without GNU parallel; one binary the action and eval already have.

### Manifest format

Extend the existing `reviewSetup` JSON emitted by `pr setup` with a new top-level field `DispatchPlan`:

```go
type DispatchPlan struct {
    Profile string          `json:"profile"`     // "fast" | "deep"
    Phases  []DispatchPhase `json:"phases"`
}

type DispatchPhase struct {
    Name          string   `json:"name"`            // "sast-triage", "analysis", "validation", "review-critical"
    Mode          string   `json:"mode"`            // "parallel" | "sequential" | "gated" | "dynamic-fanout"
    Agents        []string `json:"agents,omitempty"`
    Gate          string   `json:"gate,omitempty"`           // shell cmd that must produce non-empty output to proceed
    DynamicSource string   `json:"dynamic_source,omitempty"` // for "review-critical": yields N invocations
    DynamicAgent  string   `json:"dynamic_agent,omitempty"`  // e.g. "review-agent"
}
```

### Fast profile DispatchPlan

1. `{ name: "sast-triage", mode: "gated", gate: "zrok finding list --created-by opengrep --status open --json", agents: ["sast-triage-agent"] }`
2. `{ name: "analysis", mode: "parallel", agents: [<everything except sast-triage-agent>] }`

### Deep profile DispatchPlan

1. `{ name: "recon", mode: "sequential", agents: ["recon-agent"] }`
2. `{ name: "sast-triage", mode: "gated", ... }`
3. `{ name: "analysis", mode: "parallel", agents: [<analysis-phase agents>] }`
4. `{ name: "validation", mode: "sequential", agents: ["validation-agent"] }`
5. `{ name: "review-critical", mode: "dynamic-fanout", dynamic_source: "zrok finding list --status confirmed --severity critical --json", dynamic_agent: "review-agent" }` — fan-out computed *after* validation, not from the static manifest.

### Execution

```
zrok review pr run \
  --runner opencode|claude \
  --model <model-id> \
  --setup-json .zrok/review/setup.json \
  [--max-parallel 6]
```

Both runners support headless agent invocation with a `--agent <name>` flag:
- opencode: `opencode run --agent <name> --model <m> "<user-turn>"` reads `.opencode/agents/<name>.md`
- claude: `claude -p --agent <name> --model <m> "<user-turn>"` reads `.claude/agents/<name>.md`

Algorithm per phase (runner-agnostic):
- `parallel`: spawn `len(agents)` goroutines, each shells out to the runner-specific exec; `wait` on all; collect exit codes.
- `gated`: run gate command, parse JSON, if non-empty proceed as `sequential`.
- `sequential`: dispatch agents one at a time in declared order.
- `dynamic-fanout`: run `dynamic_source`, parse JSON list of finding IDs, for each ID call the runner with the dynamic agent and inject the finding ID into the user-turn.

Concurrency capped via `--max-parallel`. Each subprocess inherits parent env (`OPENROUTER_API_KEY`, `ANTHROPIC_API_KEY` etc. propagates). Per-agent logs to `${run_dir}/agents/<name>.log`.

The dispatcher core is one Go function with a `Runner` interface (`Spawn(agentName, model, userTurn) (*exec.Cmd, error)`). Two implementations: `opencodeRunner`, `claudeRunner`. Adding a third runner later is just a new struct.

### Coexistence with LLM-orchestrated mode (permanent — user directive)

Both modes ship and stay supported:

- **LLM-orchestrated** (today's `opencode run --agent zrok-orchestrator` or `claude -p --agent zrok-orchestrator`) — emergent / adaptive flow, agent decides what to dispatch.
- **Deterministic dispatcher** (new `zrok review pr run`) — predictable, cheaper, friendlier to small models.

`pr setup` always materializes the orchestrator agent file (no `--emit-orchestrator` flag — both paths need it). `pr run` is a NEW codepath that ignores the orchestrator and dispatches subagents directly. Users / CI workflows / the action pick which path to invoke. No deprecation, no transitional default flip.

### Eval harness changes (`eval/run.sh`)

Replace the `opencode run --agent zrok-orchestrator` call with:

```
"$ZROK_BIN" review pr run \
    --runner opencode \
    --model "$OPENCODE_MODEL" \
    --setup-json "${run_dir}/setup.json"
```

Everything downstream (finding export, manifest, scoring, baseline JSON shape) is unchanged because `zrok finding export` is what produces eval results — the dispatcher is invisible to the scorer.

### Dogfood action changes (external repo)

Coordinated PR in `diffsec/zrok-review-action`. Today the action does:

1. `zrok review pr setup --runner opencode`
2. opengrep run
3. `opencode run --agent zrok-orchestrator --model <input>`
4. `zrok review pr report`

After this change, step 3 becomes `zrok review pr run --runner opencode --model <input> --setup-json <path>`. Inputs to the action stay the same. Flag day: keep `--emit-orchestrator` true here until the action PR ships.

### Claude Code support (v1 includes this)

Two paths exist for Claude Code:

1. **SKILL.md (interactive, human-driven).** A human invokes `/code-review` in Claude Code; Claude orchestrates via its Task tool. This stays — it's the manual review path. No change for #1.
2. **Headless dispatcher (new, CI / eval / scriptable).** `zrok review pr run --runner claude` shells out `claude -p --agent <name> --model <m>` per subagent in parallel. Same `DispatchPlan` JSON drives both runners.

Required additions for claude support:
- `zrok review pr setup --runner claude` materializes `.claude/agents/*.md` (mirror of the existing `.opencode/agents/*.md` path). Frontmatter format differs slightly (Claude Code expects `description` and `prompt` fields; opencode expects `bash`, `model`, etc.) — abstract the materialization via an `agentEmitter` interface with `OpenCodeEmitter` and `ClaudeEmitter` implementations.
- `Runner` interface in dispatcher with two implementations as described in "Execution" above.
- One unit test per runner that asserts the exec command shape (without actually running it).

Open question deferred to implementation: does Claude Code's headless `--agent` flag respect the model flag for that specific agent, or does it use the agent's frontmatter `model:` if set? Need to check; affects whether per-call `--model` override is honored or silently ignored. Quick `claude -p --agent X --model Y` smoke test answers it in seconds.

### Risks

- **Per-subagent opencode spawn overhead** (~3-5s cold-start observed). Parallel goroutines hide it for parallel phases but might *increase* wall clock for fast profile if cold-start dominates. Mitigate: measure before merging; if regression is large, look at opencode session-reuse mode.
- **Subprocess auth surface.** N subprocesses each authenticate independently. Verify OpenRouter etc. don't rate-limit on parallel auth handshakes.
- **Logs proliferate.** N subprocesses = N log files. Stream each to `${run_dir}/agents/<name>.log` and print one-line per-agent summary on completion.

## Change #3: Validate + one-shot retry

### Where validation lives

Inside the new `zrok review pr run`, around each `opencode run` subprocess. Not inside `zrok finding create` — that already errors on malformed YAML and the error gets buried in opencode's stderr.

### Classification

Three categories, only the first retries:

1. **Recoverable failure → retry once:**
   - Subprocess exit non-zero AND log contains one of: `SchemaError`, `failed to parse`, `invalid yaml`, `unexpected EOF`, `tool not found`, the opencode tool-validation error patterns we already grep for.
   - Subprocess succeeded but stderr contains `zrok finding create: failed to parse`.

2. **Legitimate empty result → do NOT retry:**
   - Exit 0 AND no findings created. Valid outcome (agent looked, found nothing). Detecting "claimed N but filed 0" by parsing transcripts is fragile; don't.

3. **Hard failure → do NOT retry, surface error:**
   - Quota / rate-limit (the `is_quota_error` patterns from `eval/run.sh`).
   - Auth failure, network down.

### Retry mechanics

After category-1 detection, dispatcher constructs a corrective user-turn:

```
Your previous attempt failed: <last-100-chars-of-stderr-grep-match>.
Re-run with these corrections:
- Required Task-tool fields: description, prompt (see system prompt)
- zrok finding create requires --title, --severity, --cwe, --file, --line, --description, --confidence, --created-by
- Here is a valid invocation: <FindingCreateExample()>
Try again. This is your final retry.
```

Re-invoke `opencode run --agent <name> --model <m>` with this prompt. Agent's system prompt is unchanged (still loaded from `.opencode/agents/<name>.md`). One retry max, no exponential backoff.

### Detection helper

```go
type agentRunResult struct {
    Agent           string
    ExitCode        int
    FindingsCreated int
    StderrSnippet   string
    NeedsRetry      bool
    RetryReason     string
}
func classifyAgentRun(name string, exitCode int, stderr string, findingsBefore, findingsAfter int) agentRunResult
```

Unit-testable in isolation. Greps for known failure-mode regexes; counts findings before/after using `finding.Store.List` filtered by `created-by=<name>`.

### Eval baseline compatibility

Retry is transparent; `zrok finding export` runs after retries complete; eval scorer sees the same JSON shape. Mean precision / recall / vuln detection rate fields unchanged. Add `retries_per_agent` to the run manifest for visibility.

### Claude Code SKILL path

Unaffected. The closest analog is the SKILL's "Filing Protocol (MANDATORY)" instruction telling agents to verify their filings via `zrok finding list --created-by <self>`. Keep that — it's the SKILL path's version of #3.

## Cross-cutting risks

- **Cold-start dominance.** Confirm with timed measurement before merging #1. Cheap to verify: hack `eval/run.sh` to spawn agents in parallel via `&` and compare wall clock.
- **Subagent prompt-size growth.** Adding exemplars to every yaml grows each rendered prompt by 200-400 tokens. Watch token costs on tiny models.
- **Schema drift.** If `zrok finding create` adds required fields, `tool_examples.go` is the single update point but every agent prompt referencing it embeds the old form until rebuilt. Bitrot test in 2b catches missing markers, not stale forms. Add a stricter shell-parse test to mitigate.
