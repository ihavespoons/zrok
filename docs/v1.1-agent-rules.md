# v1.1 plan — agent-authored rules and exceptions, with a rule-noise judge

Capture of the design agreed before tagging v1. Implementation is deferred
until v1 has at least one real PR through it.

## Motivation

zrok v1 catches issues at PR time but learns nothing across PRs. Two
artifacts would change that:

- **Rules** — codify a vulnerability pattern an agent observed so it gets
  caught automatically in future PRs (additive).
- **Exceptions** — record that a specific finding or pattern is acceptable
  in this codebase so it stops re-flagging (subtractive).

Together they let the review loop accumulate org-specific knowledge over
time, with the LLM contributing without humans writing YAML by hand.

The third piece — a **judging subagent** that periodically audits the
accumulated rules — exists because every additive rule system eventually
turns into noise. Without an opposing force, rules age into false-positive
generators that erode trust in the bot.

## Artifacts

### Rules

Opengrep-format YAML at `.zrok/rules/<slug>.yaml`. Picked up automatically
by `zrok sast` via the merged `--config` path.

Each agent-authored rule carries provenance:

```yaml
# zrok-meta:
#   created_by: agent:injection-agent
#   created_at: 2026-09-15T14:23:11Z
#   created_for: PR #482
#   reasoning: |
#     Observed a pattern in app/services/billing.py where every
#     query gets a hand-built string. This idiom is repository-wide
#     and worth catching at SAST time.
rules:
  - id: zrok-org-hand-built-sql
    pattern: |
      $DB.execute($X + $Y)
    languages: [python]
    severity: ERROR
    message: Hand-built SQL string — use parameterized queries.
    metadata:
      cwe: CWE-89
      category: security
```

### Exceptions

YAML at `.zrok/exceptions.yaml`. Two shapes:

```yaml
- fingerprint: 4ce97f8a54722580f6573ef7461a2b2e4cb9426370faedd5aecbae8374298fd0
  reason: "test fixture, not production code"
  expires: 2026-09-01
  approved_by: agent:sast-triage-agent
  approved_for: PR #482

- path_glob: tests/**/*.py
  cwe: CWE-89
  reason: "test fixtures intentionally use raw SQL to exercise edge cases"
  expires: 2027-01-01
  approved_by: human:reviewer@diffsec
```

Fingerprint-keyed exceptions suppress one specific finding. Pattern-based
exceptions suppress everything matching `path_glob` × `cwe`. Both **require
an `expires:` date** — see §risks.

Honored at:
- `zrok finding list` (filtered out unless `--include-suppressed`)
- `zrok review pr report` (excluded from PR comment + SARIF)
- Audit log entry per suppression so reviewers can see what was filtered

## CLI surface

The agent **never** writes files directly. All writes go through CLI
commands that validate input, enforce required fields, and audit:

```
zrok rule add --file <yaml>
zrok rule list
zrok rule remove <slug>
zrok rule audit                        # invokes the judging subagent

zrok exception add --fingerprint <fp> --reason "..." --expires <date>
zrok exception add --path-glob "tests/**" --cwe CWE-89 --reason "..." --expires <date>
zrok exception list
zrok exception remove <id>
zrok exception expire                  # marks all expired exceptions inactive
```

This preserves the OpenCode agent permission model: agents keep
`edit: deny` / `write: deny`, the rule and exception CLIs are added to the
bash allowlist when toggled on.

## Toggle surface

Default off. Per-project config:

```yaml
# .zrok/project.yaml
allow_agent_writes:
  rules: false
  exceptions: false
```

Action input:

```yaml
- uses: diffsec/zrok-review-action@v1
  with:
    allow-agent-rules: false
    allow-agent-exceptions: false
```

When disabled, the orchestrator's prompt omits the rule/exception sections
entirely and the CLI commands aren't in the agent's bash allowlist — so
the model doesn't even know it could write.

## Judging subagent — `rule-judge-agent`

Without this, the rule set grows monotonically and slowly degrades. Same
failure mode as every accumulating linter ruleset.

### What it does

For each rule in `.zrok/rules/` and each exception in
`.zrok/exceptions.yaml`, the judge evaluates:

**Rules:**
- **Trigger count** — how many times has this rule fired across runs?
- **False-positive ratio** — of findings this rule produced, how many were
  later marked `false_positive` by the triage or validation agents?
- **Overlap** — does another rule (built-in or org-authored) catch the
  same patterns? (Run rule against a sample of historical hits, check
  fingerprint overlap.)
- **Currency** — does the rule still match anything in the current
  codebase? A rule that never fires in three months is probably dead.

Verdicts per rule, written back via `zrok rule annotate <slug>`:

| Verdict | Trigger | Action |
|---|---|---|
| `keep` | low FP ratio, fires periodically | nothing |
| `refine` | high FP ratio (>50%), fires often | suggest narrowing the pattern; agent proposes a tightened rule for human review |
| `retire` | hasn't fired in N days, or FP ratio >80%, or fully subsumed by another rule | mark `disabled: true` in the rule YAML (don't delete — keeps reasoning trail) |
| `escalate` | judge can't tell — needs a human | leave a comment with the rule's stats |

**Exceptions:**
- **Stale** — fingerprint exceptions where the code at that location has
  changed substantially since suppression (file hash drift)
- **Expiring soon** — exceptions within 14 days of `expires:` get flagged
  for renewal review
- **Over-broad** — pattern-based exceptions suppressing high-severity CWEs
  across large path globs get flagged for narrowing

### When it runs

Three modes:

1. **On-demand**: `zrok rule audit` — runs the judge against current state,
   prints a report. No mutation unless `--apply` is passed.
2. **Scheduled**: a separate GitHub Action `.github/workflows/zrok-rule-audit.yml`
   that runs the judge weekly and opens a PR with proposed changes.
3. **Threshold-triggered**: when the rule count crosses N (default 25),
   the next PR review pipeline appends a one-line note to the comment:
   *"Rule set has grown to 30 rules — consider running `zrok rule audit`."*

### Where it lives

`internal/agent/configs/agents/rule-judge-agent.yaml`. Phase: `meta`
(new — runs outside the standard review workflow). `mode: subagent`,
small-model preference, edit/write/webfetch denied, bash allowlist
restricted to `zrok rule *`, `zrok finding list`, `zrok memory read`.

### What it does NOT do

- Doesn't write new rules — that's the analysis agents' job.
- Doesn't approve/reject exceptions on first creation — that's the human
  reviewer or the sast-triage-agent's job.
- Doesn't auto-delete anything — `retire` writes `disabled: true`, leaving
  the rule and its history intact for archaeology.

## Risks and mitigations

1. **Exception drift** (Snyk Ignore failure mode).
   *Mitigation:* mandatory `expires:`; `zrok exception expire` removes
   expired ones at the start of each run; judge surfaces "expiring soon."

2. **Rule quality drift.**
   *Mitigation:* every rule carries `zrok-meta` provenance; judge runs
   every rule against historical findings and current code; retire
   threshold automated.

3. **Adversarial code triggers writes.**
   *Mitigation:* agents have no `edit`/`write` permissions; all writes go
   through validating CLI commands; every write logs to `.zrok/audit.jsonl`;
   the toggle is off by default.

4. **Same-PR feedback loop** — agent writes a rule, SAST runs it on the
   same PR, finds 80 new things.
   *Mitigation:* sequence the workflow so rule/exception writes happen
   AFTER `zrok sast` runs. The new rule applies to the next PR, not this
   one. The agent gets explicit instruction about this in its prompt.

5. **Rule audit subagent's own false-positives.**
   *Mitigation:* judge writes verdicts as annotations, not deletes. A
   wrong "retire" verdict marks `disabled: true` which is reversible.

6. **Auto-rules pollute SARIF code-scanning history.**
   *Mitigation:* opengrep rules have stable IDs; new rule IDs surface as
   "new rule firing" in code-scanning rather than "new finding everywhere."

## Open questions

These are deliberately not decided here — defer until we start building:

1. **Where do agent-authored rules live during a PR run?** Same PR
   (committed by the bot), separate PR opened by the bot, or a workflow
   artifact for a human to apply? Same-PR is simplest, separate-PR is
   safest. Recommend separate-PR for v1.1, evaluate after a quarter.

2. **Cross-repo rule sharing.** Org-wide rule sets across multiple
   repos — defer. If we build it, probably a `zrok rule pull <gh-repo>`
   that imports from a designated org rules repository.

3. **Judge model.** A small fast model handles trigger-count/overlap
   verdicts well. "Refine" verdicts (proposing a tightened pattern) want
   a stronger model. Probably two models, gated by verdict-stage.

4. **Backfill.** When v1.1 ships, do existing fingerprinted findings get
   eligible for retroactive exception authoring, or only new findings?
   Lean toward retroactive — fingerprints are stable across versions.

## Out of scope for v1.1

- ML-based rule suggestion (would require training)
- Rule effectiveness across multiple repos (cross-repo audit)
- Visual rule editor / dashboard for human-authored rules — CLI only
- IDE integration for rule writing
